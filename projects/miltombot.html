MILTOM-Bot: A Tender Bot That Earns Its Own Training Data

A Polish-language procurement-monitoring pipeline that watches the national public tender bulletin for water-tank contracts, emails them to the sales team, and captures their verdicts as labelled training data.

Back to projects

Software developer at MILTOM, a Polish manufacturer of water tanks for fire protection and drinking water. In production since 16 July 2026 — first commit 1 July 2026. The repository is private: this is a live system serving real users at a real company.

Solid = today (v0)  ·  Dashed = v1

What this is

Poland publishes every public tender to the Biuletyn Zamówień Publicznych (BZP). Somewhere in that firehose are the contracts worth bidding on — in MILTOM's case, water tanks for fire protection and drinking water. Finding them by hand means a person reading listings every morning, and people don't do that reliably for long.

The obvious move is an ML classifier. The problem is that on day one there is no training data, and no amount of cleverness conjures it. So the system is designed to earn its training data: a deliberately crude keyword filter casts a wide net, every surviving tender is emailed to the sales team with two buttons — TAK (worth pursuing) or NIE (not) — and every click is a labelled example.

v0 is a data-collection instrument disguised as a useful tool. v1 trains on what it gathers.

Why the loop works

The people best qualified to judge a tender do so in one second, as a side effect of work they were already doing. Nobody was asked to label a dataset; they were asked whether a contract is worth bidding on — which is their job. The labels are a by-product of the tool being genuinely useful, which is the only reason they keep arriving.

Architecture

Two containers that never call each other. All communication goes through Postgres.

The notifier has no inbound network surface at all — nothing can reach it. The label endpoint has no scheduler, no email, and no business logic in its route handlers. Because neither service calls the other, either can be restarted, redeployed or fail entirely without the other noticing; the database is the only contract between them.

The schedule

Three independent APScheduler jobs inside the notifier. Each runs on its own clock with its own database session.

Job Schedule Does
run_ingest hourly fetch BZP → classify → deduplicate → store
notify_salesmen Mon–Fri 08:00 send each salesperson up to 5 unseen tenders
run_admin_report daily 23:00 ops heartbeat to the admin mailbox

Ingest and notify never share a session and have no shared code path. If a send fails, no history row is written, the tender stays unsent, and tomorrow's digest retries it. Failure degrades to delay rather than loss. Deduplication is idempotent at two levels — globally (has this BZP notice ever been seen?) and per-salesperson (has this person already had this one?) — so running the pipeline twice produces the same result as running it once, which is what makes a hung run safe to simply re-trigger.

Decisions worth talking about

HTTP semantics
GET reads, POST writes — because email scanners click links

The label links go out in email. Corporate mail scanners and link-preview bots follow every URL in an inbound message before a human ever sees it. If a GET recorded a label, the scanner would silently forge one.

So GET is side-effect-free: it validates the token and renders a confirmation page with a button. The POST behind that button does the write. A bot that blindly follows links labels nothing. This is ordinary HTTP semantics, but the consequence of getting it wrong is a poisoned dataset that looks completely normal — every row plausible, no error anywhere, and a model trained on noise.

Deployment
The system had to be safe to test against production

There is one BZP API and it is the real one. Testing the pipeline end to end means real fetches — and the risk of real emails to real colleagues from a half-finished build.

Each salesperson row carries a notification mode of production, dev or both, and an environment flag selects which cohort a run targets. Development addresses receive everything; colleagues receive nothing until the flag flips. The entire stack ran against live data, sending real mail and writing real labels, for hours before a single colleague was in scope.

Label semantics
Unlabeled is unknown, not negative

A tender nobody clicked is not a "no" — it might be a "no", or it might be a busy Tuesday.

Treating silence as a negative label would train the model on the sales team's calendar rather than their judgement. Absence of evidence is stored as absence of evidence. It costs volume in the training set, and it is the difference between a model that learns what a good tender looks like and one that learns when people are at their desks.

Testing

Integration tests run against a real PostgreSQL database, and the fixture applies the real Alembic migrations rather than building the schema from the ORM models. The shortcut lets the models and the migrations silently drift apart — tests pass forever while production breaks on the next deploy. Running the real migrations makes that drift a test failure instead of an outage.

Mocking the database would make the suite faster and would test nothing that matters: constraints, cascades and transaction semantics live in Postgres, not in Python. Covered: token paths (valid, expired, tampered, wrong recipient), deduplication at both levels, classifier positives and negatives, repository read/write paths.

Operations

Docker Compose brings up Postgres, a one-shot migration container, and the two services — the app containers are gated on the migration completing successfully, so a failed migration means nothing starts rather than something starts wrong. Backups run nightly with 14-day retention, and the restore was tested into a throwaway database, verifying the row count came back. An untested backup is a file with a reassuring name.

Monitoring is one email a day. No dashboard, no Prometheus, no alerting stack, because five users and one developer do not need one. If the nightly report doesn't arrive, something is wrong. That is the entire system, and it fits in a sentence.

Silence is indistinguishable from being dead

A salesperson with no matching tenders gets a short "brak nowych przetargów" (no new tenders) email rather than nothing at all. If the bot only speaks when it has something to say, then a bot that has crashed looks exactly like a quiet week — and by the time anyone notices, weeks of tenders are gone. The same reasoning drives the nightly ops report: it is the only thing standing between a silent failure and a discovery three months later.

Roadmap

v0 — live
Keyword filter → email everything plausible → collect labels

In production since 16 July 2026. A crude filter casts a wide net; every surviving tender is emailed to the sales team, and every TAK/NIE click is stored as a labelled example.

v1 — next
TF-IDF + Logistic Regression trained on those labels → email only high-confidence

The classifier replaces the keyword filter. A hosted email sender replaces smtplib, and per-person digest timing switches on.

v2 — later
Dashboard · sources beyond BZP

Only once the classifier earns it.

The classifier is noisy today and it is supposed to be. Tightening it would reduce the volume of labels, which is the only asset v0 exists to produce. Precision is v1's job; recall is v0's. Deferring the interesting work is what makes the interesting work possible.

Known debt — scheduled, not accidental

Every one of these was a deliberate v0 trade: cheap now, replaced on a known trigger.

Email delivery

smtplib with an app password is a hard ceiling at roughly 500 messages a day. Fine for five users, and a v1 replacement is already scoped — a deliberate debt with a known limit rather than an unknown one.

Digest cadence

Timing is a single global cron, though the salesmen table already carries a per-person interval column. The column exists and is unused — a v1 hook, or premature schema, depending on how charitable you feel.

No CI

mypy --strict, ruff and the test suite are run by hand. Fine for one developer on a fifteen-day build; the first thing to add for a second.
