MILTOM-Bot: a tender-monitoring pipeline that collects its own training data

A Polish-language procurement-monitoring pipeline. It watches the national public tender bulletin for water-tank contracts, emails the matches to the sales team, and stores their verdicts as labelled training data.

Back to projects

I built this as a software developer at MILTOM, a Polish manufacturer of water tanks for fire protection and drinking water. It has been in production since 16 July 2026; the first commit was 1 July 2026. The repository is private, since it is a live system serving real users at the company.

Solid = today (v0)  ·  Dashed = v1

What this is

Poland publishes every public tender to the Biuletyn Zamówień Publicznych (BZP). A handful of those contracts are worth bidding on; for MILTOM, that means water tanks for fire protection and drinking water. Finding them by hand means someone reads the listings every morning, and that is not something people keep up reliably.

The obvious solution is an ML classifier, but on day one there is no training data to build one from. So v0 collects that data as a side effect of being useful. A deliberately crude keyword filter casts a wide net, and every tender it keeps is emailed to the sales team with two buttons: TAK (worth pursuing) or NIE (not). Each click is one labelled example.

v0 is already useful to the sales team today, and while it runs it is building the dataset that v1 will train on.

Why the loop works

The people best qualified to judge a tender are the sales team, and the click takes them about a second on work they were doing anyway. They are not being asked to label a dataset; they are being asked whether a contract is worth bidding on, which is their job. The labels come out of that, so they keep arriving as long as the tool stays useful.

Architecture

Two containers that never call each other. All communication between them goes through Postgres.

The notifier has no inbound network surface, so nothing can reach it from outside. The label endpoint has no scheduler, no email, and no business logic in its route handlers. Because neither service calls the other, either one can be restarted, redeployed, or fail without the other noticing. The database is the only contract between them.

The schedule

Three independent APScheduler jobs run inside the notifier. Each has its own clock and its own database session.

Job Schedule Does
run_ingest hourly fetch BZP → classify → deduplicate → store
notify_salesmen Mon–Fri 08:00 send each salesperson up to 5 unseen tenders
run_admin_report daily 23:00 ops heartbeat to the admin mailbox

Ingest and notify never share a session and have no shared code path. If a send fails, no history row is written, the tender stays unsent, and tomorrow's digest retries it, so a failure turns into a delay rather than a lost tender. Deduplication is idempotent at two levels: globally (has this BZP notice ever been seen?) and per salesperson (has this person already had this one?). Running the pipeline twice gives the same result as running it once, which is why a hung run is safe to re-trigger.

Design decisions

HTTP semantics
GET reads, POST writes, because email scanners click links

The label links go out by email. Corporate mail scanners and link-preview bots follow every URL in an inbound message before a human sees it. If a GET recorded a label, the scanner would forge one on its own.

So GET has no side effects: it validates the token and renders a confirmation page with a button. The POST behind that button does the write, so a bot that follows links labels nothing. This is ordinary HTTP semantics, but getting it wrong here produces a poisoned dataset that looks normal. Every row is plausible, there is no error anywhere, and the model trains on noise.

Deployment
The system had to be safe to test against production

There is one BZP API and it is the live one. Testing the pipeline end to end means real fetches, which also means a half-finished build could email real colleagues.

Each salesperson row carries a notification mode of production, dev, or both, and an environment flag selects which group a run targets. Development addresses receive everything, and colleagues receive nothing until the flag is flipped. The whole stack ran against live data, sending real mail and writing real labels, for hours before a single colleague was in scope.

Label semantics
Unlabelled means unknown, not negative

A tender nobody clicked is not a "no". It might be a "no", or it might just have been a busy Tuesday.

Treating silence as a negative label would train the model on the sales team's calendar instead of their judgement, so an unclicked tender is stored as having no label rather than a negative one. This costs some volume in the training set, but it keeps the model learning what a good tender looks like instead of when people happen to be at their desks.

Testing

Integration tests run against a real PostgreSQL database, and the fixture applies the real Alembic migrations instead of building the schema straight from the ORM models. Building from the models would let the models and the migrations drift apart, so the tests would keep passing while production broke on the next deploy. Running the real migrations turns that drift into a test failure.

Mocking the database would make the suite faster while testing nothing that matters, because the constraints, cascades, and transaction semantics live in Postgres rather than in Python. The tests cover the token paths (valid, expired, tampered, wrong recipient), deduplication at both levels, classifier positives and negatives, and the repository read and write paths.

Operations

Docker Compose brings up Postgres, a one-shot migration container, and the two services. The app containers are gated on the migration finishing successfully, so a failed migration means nothing starts rather than something starting wrong. Backups run nightly with 14-day retention, and I tested the restore into a throwaway database and checked that the row count came back. A backup nobody has restored is not really a backup.

Monitoring is one email a day. There is no dashboard and no alerting stack, because five users and one developer do not need one. If the nightly report does not arrive, something is wrong.

Why the bot mails you nothing

A salesperson with no matching tenders still gets a short "brak nowych przetargów" (no new tenders) email rather than nothing at all. If the bot only mailed when it had something to say, a crash would look identical to a quiet week, and weeks of tenders could go missing before anyone noticed. The nightly ops report exists for the same reason: without it, a silent failure could go undiscovered for months.

Roadmap

v0 — live
Keyword filter → email everything plausible → collect labels

In production since 16 July 2026. A crude filter casts a wide net, every tender it keeps is emailed to the sales team, and every TAK/NIE click is stored as a labelled example.

v1 — next
TF-IDF + Logistic Regression trained on those labels → email only high-confidence

The classifier replaces the keyword filter. A hosted email sender replaces smtplib, and per-person digest timing switches on.

v2 — later
Dashboard · sources beyond BZP

Only once the classifier is good enough to justify it.

The filter is deliberately noisy at this stage. Tightening it would cut the volume of labels, and volume is the one thing v0 exists to produce. v0 is optimised for recall, and precision is v1's job. Leaving the classifier for later is what makes it possible to train a good one at all.

Known limitations

Each of these was a deliberate v0 trade-off: cheap to build now, with a known point at which it gets replaced.

Email delivery

smtplib with an app password caps out at roughly 500 messages a day. That is fine for five users, and the v1 replacement is already scoped, so the limit is a known one rather than a surprise.

Digest cadence

Timing is a single global cron, even though the salesmen table already has a per-person interval column. The column exists but is unused for now, ready for v1 to switch on.

No CI

mypy --strict, ruff, and the test suite are run by hand. That is workable for one developer on a fifteen-day build, and it is the first thing I would add for a second developer.
